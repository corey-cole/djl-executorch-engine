# ExecuTorch Runtime Pin Bump (v1.3.1-2 → v1.3.1-6) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move `native/cmake/EtRuntimePin.cmake` from release `v1.3.1-2` to `v1.3.1-6` as a standalone, reviewable supply-chain change, landing the `windows-x86_64` row that PR 2 needs.

**Architecture:** This is a **data-only change to one file**. No CMake logic, no scripts, no Java. The pin file records a URL + SHA-256 per (variant, platform); CMake `FetchContent`s the row and re-verifies the hash on every build. Bumping it re-points every Linux artifact at newly-built tarballs and adds inert rows for `windows-x86_64` (logging) and `linux-aarch64` (all three variants) that nothing resolves yet.

**Tech Stack:** CMake 3.24+, `gh` CLI, Docker (`quay.io/pypa/manylinux_2_28_x86_64`), bash.

## Global Constraints

- Upstream release: **`v1.3.1-6`** in repo `measly-java-learning/executorch-runtime-dist`.
- ExecuTorch version is unchanged at **1.3.1** — only the *package revision* moves (`-2` → `-6`). Tarball stems (`executorch-runtime-1.3.1-<variant>-<platform>`) are therefore **identical**; only the `/download/v1.3.1-6/` path segment and the SHA-256 values change.
- The pin file header says **"Do not edit by hand"** and **"replace this whole file"**. Follow that: take the released asset verbatim, then re-apply only the repo's explanatory comment block (existing precedent — the current file already carries hand-written commentary beneath the generated header line).
- **Linux is the only platform built in this PR.** The `windows-x86_64` and `linux-aarch64` rows land unused and unresolved. Do not add matrix entries, do not uncomment `native-build-job.yml:20-24`.
- The SHA-256 diff **is** the supply-chain review gate. Never hand-type a hash; only ever copy from the released asset.

### Task 1: Bump the pin file

**Files:**
- Modify: `native/cmake/EtRuntimePin.cmake` (replace wholesale, then re-apply comment block)
- Test: `native/tests/cmake_resolution.sh:17-21`

**Interfaces:**
- Consumes: nothing.
- Produces: `ET_RUNTIME_VERSION` = `"1.3.1-6"`; `ET_RUNTIME_ET_VERSION` = `"1.3.1"` (unchanged); new variables `ET_RUNTIME_URL_logging_windows-x86_64` / `ET_RUNTIME_SHA256_logging_windows-x86_64`, plus `{bare,logging,devtools}_linux-aarch64` rows. PR 2 consumes the `windows-x86_64` row.

- [ ] **Step 1: Write the failing test**

`cmake_resolution.sh` currently asserts only the tarball *stem*, which is byte-identical across `-2` and `-6` — so it cannot detect the bump. Add a release-tag assertion so this task has a real red/green cycle.

In `native/tests/cmake_resolution.sh`, immediately after the existing default-resolution block (after line 21, the `default url wrong` check), add:

```bash
# The pin must point at the v1.3.1-6 release: the ET version (1.3.1) is unchanged from v1.3.1-2, so
# the tarball stem alone cannot distinguish the two. Assert the release-tag path segment instead.
grep -q '/download/v1.3.1-6/'                                 <<<"${out}" || fail "pin is not at release v1.3.1-6"
```

- [ ] **Step 2: Run the test to verify it fails**

Run: `bash native/tests/cmake_resolution.sh`

Expected: `FAIL: pin is not at release v1.3.1-6` (exit 1). The URL still contains `/download/v1.3.1-2/`.

- [ ] **Step 3: Fetch the released pin asset and install it**

Never hand-edit hashes. Pull the asset and overwrite the file:

```bash
gh release download v1.3.1-6 \
  --repo measly-java-learning/executorch-runtime-dist \
  --pattern 'EtRuntimePin.cmake' \
  --output native/cmake/EtRuntimePin.cmake --clobber
```

Verify it landed at the right revision and carries the Windows row:

```bash
grep -c 'ET_RUNTIME_URL_' native/cmake/EtRuntimePin.cmake   # expect 7
grep 'ET_RUNTIME_VERSION' native/cmake/EtRuntimePin.cmake   # expect "1.3.1-6"
grep 'logging_windows-x86_64' native/cmake/EtRuntimePin.cmake  # expect URL + SHA256 lines
```

- [ ] **Step 4: Re-apply the repo's comment block**

The generated asset carries only a one-line header. The repo's copy carried explanatory commentary that is still true and still worth having; re-apply it directly beneath the generated `# Generated by ...` line, matching the previous file's layout. Replace the top of `native/cmake/EtRuntimePin.cmake` so it reads:

```cmake
# Generated by executorch-runtime-dist release v1.3.1-6. Do not edit by hand.
# All three Linux variant rows are committed on purpose: `logging` is the SHIPPED/default runtime;
# `bare` and `devtools` exist ONLY for native/build_variants.sh benchmarking. Shipping only uses logging.
# windows-x86_64 ships `logging` ONLY — there is no bare/devtools Windows build upstream.
# The linux-aarch64 rows are currently INERT: no build resolves them yet (see native-build-job.yml).
# Bump procedure: replace this whole file with the EtRuntimePin.cmake asset from the next
# `v<etver>-<pkgrev>` Repo A release, then re-apply this comment block. The SHA256 change is the
# supply-chain review gate.
set(ET_RUNTIME_VERSION "1.3.1-6")
```

Leave every `set(ET_RUNTIME_URL_...)` / `set(ET_RUNTIME_SHA256_...)` line exactly as generated.

- [ ] **Step 5: Run the test to verify it passes**

Run: `bash native/tests/cmake_resolution.sh`

Expected: `PASS: cmake resolution`

- [ ] **Step 6: Review the hash diff deliberately**

This is the gate — read it, don't skim it. Confirm **every** Linux SHA-256 changed and that no hash was hand-typed:

```bash
git diff native/cmake/EtRuntimePin.cmake
```

Expected: all six Linux `SHA256` values differ from the `-2` originals; all URLs move `/download/v1.3.1-2/` → `/download/v1.3.1-6/`; seven URL rows total (3 linux-x86_64 + 3 linux-aarch64 + 1 windows-x86_64).

Cross-check two hashes against the upstream release to prove the asset wasn't tampered with in transit:

```bash
gh release download v1.3.1-6 --repo measly-java-learning/executorch-runtime-dist \
  --pattern 'executorch-runtime-1.3.1-logging-linux-x86_64.tar.gz*' --dir /tmp/pincheck --clobber

```

Expected: the `sha256sum` output matches `ET_RUNTIME_SHA256_logging_linux-x86_64` exactly.

```bash

git commit -m "deps(runtime): bump ExecuTorch runtime pin v1.3.1-2 -> v1.3.1-6
```

### Task 2: Prove the Linux build still links against the -6 artifacts

**Files:**
- Modify: none (verification only)
- Test: `native/build.sh`, `native/build_qa.sh` via the manylinux container

**Interfaces:**
- Consumes: `ET_RUNTIME_URL_logging_linux-x86_64` / `..._SHA256_...` from Task 1.
- Produces: nothing. This task exists solely to make the supply-chain risk concrete before PR 2 builds on it.

This is the whole point of splitting the pin bump out. `cmake_resolution.sh` returns at the
`ET_PRINT_RESOLUTION` seam **before** any download or `find_package`, so it proves the pin *parses* —
it proves nothing about whether the new artifacts actually link. Do that here.

- [ ] **Step 1: Build the shim against the freshly pinned runtime**

Runs in the same container CI uses, so the glibc floor and toolchain match.

```bash
curl -L -o amazon-corretto-linux-jdk.rpm \
  https://corretto.aws/downloads/latest/amazon-corretto-8-x64-linux-jdk.rpm
docker run --rm -v "$PWD":/workspace -w /workspace \
  quay.io/pypa/manylinux_2_28_x86_64 /bin/bash /workspace/native/build.sh
```

Expected: configure downloads `executorch-runtime-1.3.1-logging-linux-x86_64.tar.gz` from
`/download/v1.3.1-6/`, the URL_HASH check passes, the build completes, and the run ends with
`Artifact: src/main/resources/native/linux-x86_64/libexecutorch_djl.so`.

A `URL_HASH` mismatch here means the pin and the artifact disagree — stop and re-verify Task 1 Step 6.

- [ ] **Step 2: Confirm the XNNPACK post-link guard passed**

The guard runs as a POST_BUILD step, so Step 1 succeeding already implies it passed. Make that explicit
rather than assumed — it is the assertion most likely to catch a bad upstream re-roll:

```bash
docker run --rm -v "$PWD":/workspace -w /workspace \
  quay.io/pypa/manylinux_2_28_x86_64 /bin/bash -c \
  'nm /workspace/native/build/libexecutorch_djl.so | grep -c "_GLOBAL__sub_I_XNNPACKBackend"'
```

Expected: `1` or greater. `0` means the `-6` tarball stopped self-whole-archiving `xnnpack_backend` —
a genuine upstream regression. Stop and escalate; do not proceed to PR 2.
